아키텍처 방향 선택

두 가지 현실적인 접근법 — MVP 속도 vs 장기 유연성

A

Supabase 중심 (추천) MVP 최적

📱 Expo 앱 ────────────────────── 🗄️ Supabase
                        └ DB (PostgreSQL)
                        └ Auth (소셜 로그인)
                        └ Storage (이미지)
⚙️ FastAPI ────────────────────── 🤖 크롤러
  (검색/필터 API만)           주 2~3회 GitHub Actions
  • Auth, DB, 실시간 구독 → Supabase가 다 처리
  • FastAPI는 검색/크롤링 로직만 담당
  • 크롤러는 GitHub Actions로 무료 스케줄링
  • 소셜 로그인(카카오·구글) 바로 연동 가능

장점

  • 가장 빠르게 출시
  • 월 비용 최소 (Supabase 무료 티어)
  • 인증 직접 구현 불필요

단점

  • Supabase 의존성
  • 트래픽 급증 시 비용 발생
B

FastAPI 풀스택 확장 현재 코드 베이스 유지

📱 Expo 앱 ────────────────────── ⚙️ FastAPI
                        └ JWT 인증 직접 구현
                        └ PostgreSQL (Railway)
                        └ APScheduler (크롤러 내장)
  • 현재 FastAPI 코드 그대로 확장
  • SQLite → PostgreSQL만 교체
  • 크롤러를 백엔드 안에서 스케줄 실행
  • JWT 인증 직접 구현 필요

장점

  • 외부 의존성 최소
  • 코드 전체 컨트롤 가능

단점

  • 인증 직접 구현 (1~2주 추가)
  • 서버 관리 부담 증가
  • 소셜 로그인 연동이 복잡

추천: A — 인증·DB·스토리지를 Supabase에 맡기면 핵심인 크롤링 파이프라인과 앱 개발에 집중할 수 있어요. 나중에 트래픽이 커지면 B 방향으로 마이그레이션도 가능합니다.
